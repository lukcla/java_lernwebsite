Einstieg in Java

Zusammenfassung

Java ist eine streng objektorientierte, explizit typisierte Sprache. Im Unterschied zu Python muss jede Variable vor ihrer Nutzung mit einem festen Typ deklariert werden. Jedes Java-Programm besteht aus mindestens einer Klasse; der Einstiegspunkt ist immer public static void main(String[] args). Der Quelltext wird mit javac zu Bytecode kompiliert und dann durch die Java-VM mit java ausgeführt.

Installationsvoraussetzungen – das JDK

Quelle: Skript Teil 1, Abschnitt „Installationsvoraussetzungen”, S. 11–13

Zum Entwickeln von Java-Programmen wird das Java Development Kit (JDK) benötigt. Es enthält unter anderem:

  • javac – der Java-Compiler: wandelt den Quelltext (.java) in maschinenunabhängigen Bytecode (.class) um.
  • java – die Java-Runtime-Environment (JRE): führt den kompilierten Bytecode in der Java Virtual Machine (JVM) aus.

Das aktuelle OpenJDK ist frei erhältlich. Für komfortables Entwickeln empfiehlt sich die IDE Eclipse (eclipse.org), die Editieren, Kompilieren und Starten in einer Oberfläche vereint.

Programmstruktur – Python vs. Java

Quelle: Skript Teil 1, Abschnitt „Programmstruktur”, S. 14–22

In Python genügt eine Abfolge von Anweisungen. In Java muss jeder Code in eine Klasse eingebettet sein. Blöcke werden mit geschweiften Klammern {} begrenzt – Einrückung ist in Java konventionell, aber nicht syntaktisch erzwungen (anders als in Python).

// Dateiname muss exakt dem Klassennamen entsprechen: Hello.java
public class Hello {
    // Einstiegspunkt des Programms – immer diese Signatur
    public static void main(String[] args) {
        System.out.println(”Hello World!”);
    } // end method main()
} // end class Hello

Wichtige Regeln:

  • Die Programmdatei muss genauso heißen wie die Klasse (Groß-/Kleinschreibung beachten) und auf .java enden.
  • Mindestens eine Klasse muss eine main()-Methode besitzen – sie ist der Startpunkt des Programms.
  • main() bekommt immer ein String[]-Array übergeben, das eventuelle Kommandozeilenparameter enthält.

Programm-Erzeugung und -Start

Quelle: Skript Teil 1, Abschnitte „Kommandozeile” & „Eclipse”, S. 23–45

Auf der Kommandozeile

// 1. Schritt: Kompilieren (erzeugt Hello.class)
javac Hello.java

// 2. Schritt: Ausführen (Klassenname OHNE .class angeben!)
java Hello

// Mehrere Quelldateien auf einmal kompilieren:
javac *.java
  • javac prüft die Syntax und erzeugt bei Erfolg die .class-Datei; bei Fehlern erscheint eine Meldung mit Zeilennummer.
  • java NameDerMainKlasse startet das Programm in der JVM.

In Eclipse

  • Programme werden in Projekten organisiert, die in einem Workspace-Verzeichnis liegen.
  • Beim Speichern einer .java-Datei wird automatisch kompiliert. Syntaxfehler erscheinen sofort im Editor.
  • Das Starten erfolgt mit dem grünen Start-Pfeil (Run). Eclipse sucht automatisch nach der Klasse mit main().

Die Java-API

Quelle: Skript Teil 1, Abschnitt „Java-API”, S. 46–49

API steht für Application Programming Interface – vorgefertigte Bibliotheken, die Entwicklern fertige Klassen und Algorithmen zur Verfügung stellen, ohne dass man alles selbst implementieren muss.

Die offizielle Java-API-Dokumentation ist eine HTML-Seite, die alle Standardklassen, ihre Methoden und deren Parameter beschreibt. Für eigene Klassen kann mit dem Tool javadoc eine solche Dokumentation automatisch erzeugt werden.

Prüfungshinweis „Schauen Sie in die API!” ist ein zentraler Rat des Skripts. Viele typische Aufgaben – Stringverarbeitung, Dateioperationen, Collections – sind bereits in der Java-API gelöst. Man muss die Klassen und Methoden kennen, nicht auswendig lernen, aber wissen, wo man nachschlägt.

Variablen, Datentypen und Operatoren

Quelle: Skript Teil 1, Abschnitt „Variablen, Datentypen und Operatoren”, S. 50–107

Elementare Datentypen & final

Java ist explizit typisiert: jede Variable muss vor ihrer Nutzung mit einem Datentyp deklariert werden (int x = 5;). Die wichtigsten elementaren (primitiven) Datentypen:

Typ Größe Wertebereich / Beschreibung Beispiel
byte8 Bit−128 … 127byte b = 2;
short16 Bit−32768 … 32767short s = 45;
int32 Bitca. ±2 Mrd.int i = 5;
long64 Bitca. ±9,2 · 10¹⁸long l = 1000L;
float32 BitIEEE 754 Gleitkommafloat f = 48.7f;
double64 BitIEEE 754, höhere Genauigkeitdouble d = 55.7;
char16 BitUnicode-Zeichen 0 … 65535char c = 'A';
boolean–true oder falseboolean ok = true;

Mit dem Modifier final wird eine Variable zur Konstante: nach der ersten Zuweisung kann kein neuer Wert mehr gesetzt werden. Ein Versuch führt zum Compilerfehler:

final double PI = 3.141592653589793;
PI = 9.0; // Compilerfehler! PI ist final.

Referenzdatentypen: Arrays, mehrdimensionale Arrays, Strings, Klassen

Aggregierte (zusammengesetzte) Datentypen wie Arrays, Klassen und Strings sind Referenztypen: die Variable speichert nicht die Daten selbst, sondern eine Referenz (Speicheradresse) auf den Bereich, in dem die eigentlichen Daten liegen.

// Array – Möglichkeit 1: new
int[] myArray = new int[4];
myArray[0] = 7; myArray[1] = -99; myArray[2] = 102; myArray[3] = 8;

// Array – Möglichkeit 2: Literal
int[] myArray2 = {7, -99, 102, 8};

// Länge: array.length (Variable, keine Methode!)
System.out.println(myArray.length); // 4

// Zweidimensionales Array (Matrix)
int[][] matrix = new int[3][5];
matrix[2][3] = 255;

Die null-Referenz bedeutet: Variable ist noch nicht mit einer gültigen Adresse initialisiert. Wird eine nicht-initialisierte Referenz dereferenziert, entsteht zur Laufzeit eine NullPointerException:

int[] array = null;
if (array != null) {
    array[0] = -99; // sicher: erst testen, dann zugreifen
}

Operatoren & Typkonvertierung – Python vs. Java

Arithmetik- (+ - * / %), Vergleichs- (== != < > <= >=) und Logik-Operatoren (&& || !) sind fast identisch mit Python. Neu in Java: Inkrement/Dekrement-Operatoren:

i++;   // Post-Inkrement: erst i verwenden, dann hochzählen
++i;   // Pre-Inkrement:  erst hochzählen, dann i verwenden
j--;   // Dekrement analog

Typkonvertierung (Typecast): Java konvertiert automatisch von „kleinem” zu „großem” Typ (impliziter Cast, z. B. int → long). In die andere Richtung muss explizit gecastet werden:

short a = 50, b = 70;
short ergebnis = (short) (a * b); // expliziter Cast: int-Ergebnis → short
Prüfungshinweis Operationen auf byte, short oder char liefern immer ein int-Ergebnis! Das direkte Zuweisen an eine short-Variable führt zum Compilerfehler, solange kein expliziter Cast geschrieben wird.

Sichtbarkeit von Variablen – Teil 1

In Java gibt es drei Gültigkeitsbereiche (Scopes):

  • Lokale Variablen: nur innerhalb der Methode/des Blocks sichtbar, in dem sie definiert wurden.
  • Instanzvariablen (Objektvariablen): innerhalb aller Methoden eines Objekts sichtbar; jedes Objekt hat seine eigene Instanz dieser Variable.
  • Klassenvariablen (static): existieren einmal für die gesamte Klasse, unabhängig davon, wie viele Objekte erzeugt wurden.

Wenn eine lokale Variable den gleichen Namen wie eine Instanzvariable hat, überlagert sie diese. Mit this.variablenname kann explizit auf die Instanzvariable zugegriffen werden.

Kontrollstrukturen

Quelle: Skript Teil 1, Abschnitt „Kontrollstrukturen”, S. 108–147

Fallunterscheidungen

// if – else if – else: für komplexere Bedingungen & Wertebereiche
if (wert >= 0 && wert <= 5) {
    System.out.println(”Zwischen 0 und 5”);
} else if (wert >= 6 && wert <= 10) {
    System.out.println(”Zwischen 6 und 10”);
} else {
    System.out.println(”Außerhalb!”);
}

// switch-case: für konstante Einzelwerte (seit Java 7 auch Strings)
switch (wert) {
    case 1: System.out.println(”Eins”); break;
    case 2: System.out.println(”Zwei”); break;
    default: System.out.println(”Anderes”); break;
}

Schleifen

// while: kopfgesteuert – Bedingung wird VOR dem ersten Durchlauf geprüft
while (zahl >= 1 && zahl <= 10) { zahl = eingabe.nextInt(); }

// for als Zählschleife
for (int i = 1; i <= 10; i++) { System.out.println(i); }

// for als Iterationsschleife (foreach) über ein Array oder eine Liste
for (int wert : zahlenArray) { System.out.println(wert); }

// do-while: fußgesteuert – Rumpf wird MINDESTENS EINMAL ausgeführt
do {
    zahl = eingabe.nextInt();
} while (zahl >= 1 && zahl <= 10);
Prüfungshinweis Im switch-Block fehlendes break führt zum Fall-through: der Code der folgenden Cases wird ebenfalls ausgeführt, bis ein break oder das Ende des switch erreicht wird! Dieses Verhalten ist manchmal gewollt (Wertebereiche), aber oft ein Bug. Die do-while-Schleife eignet sich immer dann, wenn der Schleifenrumpf mindestens einmal durchlaufen werden soll (z. B. Eingabevalidierung).

Methoden in Java

Quelle: Skript Teil 1, Abschnitt „Methoden in Java”, S. 148–161

Methoden sind in sich abgeschlossene Unteraufgaben innerhalb einer Klasse. In Java können Funktionen nur innerhalb von Klassen definiert werden – daher heißen sie hier Methoden. Die Typisierung von Parametern und Rückgabewert ist immer explizit.

// Methode ohne Rückgabe: Rückgabetyp void
void sagHallo(String name) {
    System.out.println(”Hallo “ + name);
}

// Methode mit Rückgabe: Rückgabetyp und return-Statement
int altersBerechnung(int alter) {
    int ergebnis = alter - 18;
    return ergebnis;
}

Die Signatur einer Methode ist der Methodenkopf ohne Rückgabetyp (z. B. sagHallo(String name)). Sie wird beim Überladen von Methoden benötigt.

Parameterübergabe:

  • Call-by-value (elementare Typen): Es wird eine Kopie des Wertes übergeben. Die Originalvariable in der aufrufenden Methode bleibt unverändert.
  • Call-by-reference (Referenztypen): Es wird die Adresse des Objekts übergeben. Beide Methoden arbeiten mit demselben Objekt auf dem Heap.

Einstieg in die OOP: Klassen und Objekte/Instanzen

Zusammenfassung

Objekte sind konkrete Ausprägungen (Instanzen) einer Klasse. Klassen sind Baupläne, die Attribute (Zustand) und Methoden (Verhalten) von Objekten beschreiben. Objekte werden mit new im Heap erzeugt; die Variable enthält nur die Referenz (Adresse) auf das Objekt. Das Prinzip des Information Hiding schreibt vor, Attribute als private zu deklarieren und den Zugriff über öffentliche getter/setter-Methoden zu kontrollieren.

Objekte in der OOP

Quelle: Skript Teil 1, Abschnitt „Objekte in der OOP”, S. 164

Die objektorientierte Programmierung modelliert die reale Welt in Form von Objekten. Jedes Objekt hat:

  • Attribute – beschreiben den Zustand (z. B. Farbe, Name, Wert).
  • Methoden – beschreiben das Verhalten (z. B. fahren, drucken, berechnen).

Beispiele für Objekte: ein grün lackierter Mini, ein Buch mit Titel „Grauen der OOP”, ein Besatzungsmitglied namens Spock. Die Klasse „Buch” wäre der Bauplan; ein konkretes Buch ist eine Instanz dieses Bauplans.

Klassen in Java: Vereinbarung, Konstruktoren, Objekterzeugung

Quelle: Skript Teil 1, Abschnitte „Klassen in Java” & „Vereinbarung von Klassen”, S. 165–183
public class Buch {
    // Attribute (private: kein Direktzugriff von außen)
    private String autor;
    private String titel;
    private int    auflage;

    // Standardkonstruktor (parameterlos) – setzt Standardwerte
    public Buch() {
        autor   = ”Hugo Helmchen”;
        titel   = ”Allein im Weltraum”;
        auflage = 1;
    }

    // Voll qualifizierter Konstruktor – setzt alle Attribute via Parameter
    public Buch(String autor, String titel, int auflage) {
        // this = Referenz auf dieses Objekt (wie self in Python)
        this.autor   = autor;
        this.titel   = titel;
        this.auflage = auflage;
    }

    // getter-Methode – Attribut lesen
    public String getAutor()  { return autor; }
    public String getTitel()  { return titel; }
    public int    getAuflage(){ return auflage; }

    // setter-Methode – Attribut (kontrolliert) ändern
    public void setAuflage(int auflage) { this.auflage = auflage; }
} // end class
// Objekterzeugung mit new – Objekt liegt auf dem Heap
Buch b1 = new Buch(”G. Grusel”, ”Grauen der OOP”, 5);
Buch b2 = new Buch(); // Standardkonstruktor

// Zugriff über Punkt-Operator
String aut = b1.getAutor();
b1.setAuflage(6);
b1.getAuflage(); // gibt 6 zurück
Prüfungshinweis Konstruktoren haben keinen Rückgabetyp – nicht einmal void! Sie heißen genau wie die Klasse. Wird kein expliziter Konstruktor definiert, stellt Java automatisch einen parameterlosen Standard-Konstruktor bereit (der aber entfällt, sobald ein eigener Konstruktor geschrieben wird).

Sichtbarkeit: private, public, protected, package

Quelle: Skript Teil 1, Abschnitt „Sichtbarkeit von Variablen und Methoden”, S. 185–203

Sichtbarkeitsmodifizierer steuern, welche Klassen auf Attribute und Methoden eines Objekts zugreifen dürfen. Das Prinzip des Information Hiding schreibt vor, Attribute immer private und nur die benötigte Schnittstelle public zu machen:

Modifier Sichtbar für …
publicAlle Klassen überall (auch in anderen Packages)
privateNur innerhalb der eigenen Klasse – von außen vollständig verborgen
protectedEigene Klasse + Kindklassen + Klassen im gleichen Package
(kein Modifier)Nur Klassen im gleichen Package (Package-Sichtbarkeit)

Best Practice: Attribute immer private; Getter und Setter public. So kann die interne Implementierung geändert werden, ohne dass äußerer Code angepasst werden muss.

Instanzvariablen/-methoden vs. Klassenvariablen/-methoden (static)

Quelle: Skript Teil 1, Abschnitt „Instanzvariablen vs. Klassenvariablen”, S. 204–218

Instanzvariablen/-methoden gehören zu einem konkreten Objekt. Bei 3 erzeugten Objekten existieren 3 eigene Kopien der Instanzvariablen. Zugriff immer über eine Objektreferenz: obj.methode().

Klassenvariablen/-methoden sind mit dem Modifier static gekennzeichnet. Sie existieren einmal für die gesamte Klasse, egal wie viele Objekte erzeugt werden. Zugriff direkt über den Klassennamen: KlassenName.variable.

// Sinnvolles Beispiel: statische Konstante + statische Methode
public class StaticOk {
    static final double KELVIN_NULL = -273.15; // Klassenkonstante

    static double gradCelsius(double kelvin) { // Klassenmethode
        return kelvin + KELVIN_NULL;
    }
}

// Aufruf direkt über Klassennamen – kein Objekt nötig:
double c = StaticOk.gradCelsius(300.0);
Prüfungshinweis Statische Methoden können nicht auf Instanzvariablen zugreifen! Sie kennen kein this. Das macht Sinn: eine Klassenmethode läuft ohne konkretes Objekt – welches this sollte es referenzieren?

Arrays und Listen aus Objekten/Instanzen

Quelle: Skript Teil 1, Abschnitte „Arrays aus Objekten” & „Listen aus Objekten”, S. 219–232

Objekt-Arrays – drei Schritte:

// 1. Array-Referenz erzeugen (enthält noch keine Objekte!)
Person[] personen = new Person[3];

for (int i = 0; i < personen.length; i++) {
    String name = tastatur.nextLine();
    Person p    = new Person(name); // 2. Instanz erzeugen
    personen[i] = p;                // 3. Ins Array einhängen

ArrayList – dynamisch wachsend, nur Referenztypen (keine primitiven Typen):

import java.util.ArrayList;

ArrayList<Person> liste = new ArrayList<Person>();

// Einfügen
liste.add(new Person(”Erna”));
liste.add(new Person(”Hugo”));

// Zugriff per Index (wie Array) oder foreach
Person p = liste.get(0);
for (Person x : liste) { System.out.println(x.getName()); }

Die Klasse String

Zusammenfassung

Die Klasse String kapselt Zeichenfolgen aus 16-Bit-Unicode-Zeichen. Strings sind in Java unveränderliche Objekte (immutable): jede scheinbare Änderung erzeugt intern ein neues String-Objekt. Der ==-Operator vergleicht nur Adressen, nicht Inhalte – zum inhaltlichen Vergleich muss equals() verwendet werden. Die Klasse String bietet eine umfangreiche API zum Suchen, Zuschneiden und Konvertieren von Zeichenketten.

Grundlagen: char und String

Quelle: Skript Teil 1, Abschnitt „Grundlagen Strings”, S. 235–236

Der primitive Datentyp char ist 16 Bit groß und kann ein einzelnes Unicode-Zeichen aufnehmen (Wertebereich 0–65535; das Zeichen 'A' hat z. B. den Wert 65, wie in ASCII). Zeichenliterale stehen in einfachen Anführungszeichen.

Die Klasse String speichert Zeichenfolgen und stellt viele Methoden zur Verarbeitung bereit. String-Literale stehen in doppelten Anführungszeichen:

char x = 'M';               // einzelnes Zeichen (einfache Anführungszeichen)
String s = ”Erna Etepetete”; // Zuweisung eines String-Literals
String t = new String(”Hugo Helmchen”); // Konstruktor-Variante

Stringoperationen & wichtigste Methoden

Quelle: Skript Teil 1, Abschnitte „Stringoperationen” & „Wichtigste Methoden der Klasse String”, S. 237–260

Konkatenation

String s1 = ”Hallo “;
String s2 = ”Welt”;
String s3 = s1 + s2 + 2; // “Hallo Welt2” – + verkettet Strings
s1 += s2;                 // Kurzschreibweise für s1 = s1 + s2

Vergleich – equals() statt ==

String s = ”Erna”;
String t = new String(”Erna”);

// FALSCH: == vergleicht nur Adressen!
if (s == t)      { ... } // liefert false, obwohl Inhalt gleich ist

// RICHTIG: equals() vergleicht den Inhalt
if (s.equals(t)) { ... } // liefert true

// Alternative: switch-case (seit Java 7 mit String erlaubt)
switch (t) {
    case ”Erna”: System.out.println(”Hallo Erna!”); break;
}
Prüfungshinweis == bei Strings ist ein klassischer Fehler in Java! Wenn zwei String-Literale denselben Wert haben, kann == zufällig true liefern (Java hält gleichartige Literale im Speicher-Pool), aber sobald ein String per new erzeugt wird, verweist er auf eine andere Adresse → == ergibt false. Immer equals() verwenden!

// indexOf: 'A' (Großbuchstabe) kommt nicht vor → −1
int pos1 = s1.indexOf('A');              // −1
// “dach” ab Position 5 → erste Fundstelle ab Index 5 ist 10
int pos2 = s1.indexOf(”dach”, 5);        // 10
// Teilstring zwischen Index 2 und vor Index 6: “im F”

// Konvertierung: int ↔ String

Eigenschaften von Strings abfragen

Quelle: Skript Teil 1, Abschnitt „Eigenschaften von Strings abfragen”, S. 249–251

Zur Überprüfung von String-Eigenschaften stehen u. a. folgende Methoden zur Verfügung:

String s = new String(”Hallo”);

s.length();           // 5 – Länge (Methode, nicht .length wie bei Arrays!)
s.isEmpty();          // false – ist leer?
s.charAt(1);          // 'a' – Zeichen an Position 1
s.startsWith(”Ha”);  // true
s.endsWith(”lo”);    // true
s.equals(”Hallo”);  // true – inhaltlicher Vergleich
Prüfungshinweis Die Länge eines Strings wird mit s.length() (Methode, mit runden Klammern) abgefragt. Die Länge eines Arrays mit arr.length (Variable, ohne runde Klammern). Dieser Unterschied ist ein häufiger Fehler in Klausuren!

UML-Klassendiagramme – Teil 1

Zusammenfassung

UML (Unified Modeling Language) ermöglicht die grafische Darstellung von Klassen und deren Beziehungen – kompakter und übersichtlicher als reiner Java-Code. Ein UML-Klassendiagramm zeigt in drei Blöcken: Klassenname, Attribute (mit Typ und Sichtbarkeit) und Methoden (mit Parametern und Rückgabetyp). Beziehungen zwischen Klassen – Assoziation, Aggregation, Komposition – werden durch verschiedene Linientypen und Pfeilspitzen dargestellt.

Einfache Klassendarstellungen

Quelle: Skript Teil 1, Abschnitt „Einfache Klassendarstellungen”, S. 263–270

Eine Klasse wird als Rechteck mit drei Blöcken dargestellt:

  • Oben: Klassenname
  • Mitte: Attribute – Format: Sichtbarkeit name: Typ
  • Unten: Methoden und Konstruktoren – Format: Sichtbarkeit methode(param: Typ): RückgabeTyp

Sichtbarkeiten werden als Symbole geschrieben: + = public, - = private, # = protected. Achtung: In UML steht der Typ nach dem Bezeichner (getrennt durch Doppelpunkt) – anders als in Java.

// UML-Notation (vereinfacht als Text dargestellt):
// ┌──────────────────────────────────────────────┐

In einer reduzierten Darstellung können Konstruktoren, getter und setter weggelassen werden – dann erscheinen nur die „echten” fachlichen Methoden im unteren Block.

Klassenbeziehungen: Assoziation, Aggregation, Komposition

Quelle: Skript Teil 1, Abschnitt „Klassenbeziehungen – Teil 1”, S. 271–281

Assoziation

Die allgemeinste Beziehung: Objekte zweier Klassen kennen sich gegenseitig oder einseitig. Dargestellt als Linie, oft mit Pfeilspitze (Richtung des Zugriffs) und Multiplizitäten an den Enden:

  • 1 – genau ein Objekt
  • 1..* – ein oder mehrere Objekte
  • 0..* – null oder mehrere (optional)

Beispiel: Ein Leser liest zu einer Zeit genau 1 Buch (1:1). Ein Leser kann 0 bis viele Bücher besitzen (1:0..*). Die offene Pfeilspitze zeigt, wer auf wen Zugriff hat; ein Dreieck über dem Assoziationsnamen gibt die Leserichtung an.

Aggregation vs. Komposition

Beide drücken eine „ist Bestandteil von”-Beziehung aus. Der Unterschied liegt im Lebenszyklus:

Beziehungstyp Symbol Lebenszyklus des Teils Beispiel
Aggregation Offene Raute am Ganzen Teil existiert weiter, wenn das Ganze gelöscht wird Auto – Räder (Räder können weiterverkauft werden)
Komposition Gefüllte Raute am Ganzen Teil wird mitgelöscht, wenn das Ganze gelöscht wird Gebäude – Räume (Räume gibt es nur als Teil des Gebäudes)
Prüfungshinweis Aggregation und Komposition unterscheiden sich am Lebenszyklustest: Kann das Teil auch ohne das Ganze existieren? Ja → Aggregation (offene Raute). Nein → Komposition (gefüllte Raute). Weitere Beispiele: Button in Window = Komposition; Maschine in Fabrik = Aggregation.

Vererbungsbeziehungen

Zusammenfassung

Vererbung ermöglicht die Weitergabe von Attributen und Methoden einer Superklasse (Elternklasse) an Subklassen (Kindklassen). Java unterstützt nur Einfachvererbung (eine Elternklasse pro Klasse), erlaubt aber das Implementieren mehrerer Interfaces. Durch Polymorphie kann eine Variable vom Typ der Elternklasse Objekte von Kindklassen halten – welche Methode zur Laufzeit aufgerufen wird, bestimmt der tatsächliche Objekttyp (dynamische Bindung). Abstrakte Klassen und Interfaces erzwingen die Implementierung bestimmter Methoden in den Subklassen.

Wozu Vererbung?

Quelle: Skript Teil 1, Abschnitt „Wozu Vererbung?”, S. 285–295

Ohne Vererbung müssen gemeinsame Eigenschaften (z. B. vorname, nachname) in jeder Klasse neu implementiert werden – das verstößt gegen das DRY-Prinzip (Don't Repeat Yourself). Außerdem lassen sich ist-ein-Beziehungen nicht modellieren: ein/e Studierende/r ist eine Person mit zusätzlichen Merkmalen.

Mit Vererbung wird gemeinsames Verhalten einmal in der Superklasse implementiert. Kindklassen erben dieses Verhalten automatisch und ergänzen es um ihre eigenen Besonderheiten:

  • Generalisierung: Gemeinsame Eigenschaften werden in die Superklasse „hochgezogen”.
  • Spezialisierung: Kindklassen fügen eigene Attribute/Methoden hinzu.

Einfache Vererbung: extends, Eltern-/Kindklassen, Objekterzeugung

Quelle: Skript Teil 1, Abschnitt „Einfache Vererbungsbeziehungen in Java”, S. 296–310

Die Vererbungsbeziehung wird in Java mit extends deklariert. Im Konstruktor der Kindklasse muss als erstes Statement der Superklassen-Konstruktor mit super() aufgerufen werden:

public class Person {
    private String vorname, nachname;
    public Person(String vorname, String nachname) {
        this.vorname = vorname; this.nachname = nachname;
    }
    public String getNachname() { return nachname; }
    public String getVorname()  { return vorname; }
}

public class Studierende extends Person {   // erbt von Person
    private int matrikelnr;

    public Studierende(String vorname, String nachname, int matrikelnr) {
        super(vorname, nachname); // MUSS als erstes kommen!
        this.matrikelnr = matrikelnr;
    }
    public int getMatrikelnr() { return matrikelnr; }
    // getNachname() und getVorname() sind geerbt – keine erneute Implementierung nötig!
}
Prüfungshinweis Kindklassen können nicht direkt auf private-Attribute der Elternklasse zugreifen – auch wenn sie geerbt wurden. Der Zugriff ist nur über den Superklassen-Konstruktor (super()) oder protected/public-Methoden möglich. Java unterstützt keine Mehrfachvererbung: eine Klasse kann nur von genau einer anderen Klasse erben.

Polymorphie – Vielgestaltigkeit von Objekten

Quelle: Skript Teil 1, Abschnitte „Polymorphie” & „Dynamische Bindung”, S. 308–320

Eine Variable vom Typ der Superklasse kann ein Objekt irgendeiner Subklasse aufnehmen – denn eine Studierende ist eine Person:

// Statischer Typ: Person | Dynamischer Typ: Studierende
Person s = new Studierende(”Erna”, ”Etepetete”, 4711);

// OK: getNachname() ist in Person bekannt
String name = s.getNachname();

// FEHLER: getMatrikelnr() kennt der statische Typ Person nicht!
int matnr = s.getMatrikelnr(); // Compilerfehler

// Lösung: Downcast auf den konkreten Typ
if (s instanceof Studierende) {
    Studierende stud = (Studierende) s;
    int nr = stud.getMatrikelnr();
}

Dynamische Bindung: Wenn eine Methode in der Kindklasse überschrieben wurde, entscheidet zur Laufzeit der dynamische Typ (der tatsächliche Objekttyp), welche Methodenimplementierung aufgerufen wird – nicht der statische Typ der Variable.

Prüfungshinweis Klassische Klausuraufgabe: Eine Variable vom Typ Person zeigt auf ein Studierende-Objekt. Beide Klassen haben eine Methode print(). Welche wird aufgerufen? → Die der Kindklasse (Studierende), weil Java die Methode dynamisch zur Laufzeit bindet. Der Compiler schaut auf den statischen Typ; die JVM beim Ausführen auf den tatsächlichen Objekttyp.

Die Superklasse aller Klassen: Object

Quelle: Skript Teil 1, Abschnitt „DIE Superklasse: Object”, S. 321–330

Alle Java-Klassen erben implizit von der Klasse java.lang.Object. Dadurch stehen in jeder eigenen Klasse bestimmte Methoden zur Verfügung, die bei Bedarf überschrieben werden können:

  • toString() – gibt eine textuelle Beschreibung des Objekts zurück. Standard: Klassenname + Hashcode. Sollte in eigenen Klassen überschrieben werden, um sinnvolle Ausgaben zu erhalten.
  • equals(Object o) – vergleicht zwei Objekte auf Inhalt. Standard: Referenzvergleich (wie ==). In eigenen Klassen überschreiben, um echten Inhaltsvergleich zu ermöglichen.
  • hashCode() – Hash-Wert des Objekts; muss konsistent mit equals() sein.
// toString() überschreiben
@Override
public String toString() {
    return ”Person[“ + vorname + ” “ + nachname + ”]”;
}

Überladen vs. Überschreiben

Quelle: Skript Teil 1, Abschnitt „Überladen vs. Überschreiben”, S. 331–345
Konzept Wo? Signatur Zweck
Überladen (Overloading) Gleiche Klasse Gleicher Methodenname, andere Parameter Mehrere Varianten einer Methode für verschiedene Eingaben
Überschreiben (Overriding) Kind- vs. Elternklasse Identische Signatur Kindklasse ersetzt die geerbte Implementierung durch eigene 
// Überladen: gleicher Name, andere Parameterliste
void print() { System.out.println(name); }
void print(String prefix) { System.out.println(prefix + name); }

// Überschreiben: identische Signatur in Kindklasse
// In Superklasse Person:
public String toString() { return ”Person: “ + name; }

// In Kindklasse Studierende:
@Override
public String toString() { return ”Stud: “ + name + ” (“ + matnr + ”)”; }

Abstrakte Klassen & @Override

Quelle: Skript Teil 1, Abschnitte „Abstrakte Klassen” & „Einschub: @Override”, S. 346–370

Eine abstrakte Klasse ist ein unvollständiger Bauplan: sie kann abstrakte Methoden enthalten, die keinen Methodenrumpf haben. Abstrakte Klassen können nicht instanziiert werden – sie erzwingen, dass Kindklassen die abstrakten Methoden implementieren.

public abstract class Tier {
    private String name;

    public Tier(String name) { this.name = name; }

    // Abstrakte Methode: kein Rumpf, muss in Kindklasse implementiert werden
    public abstract void lautGeben();

    // Normale Methode: vererbt wie gewohnt
    public String getName() { return name; }
}

public class Hund extends Tier {
    public Hund(String name) { super(name); }

    @Override            // Annotation: prüft, ob tatsächlich überschrieben wird
    public void lautGeben() { System.out.println(”Wuff!”); }
}

Die Annotation @Override ist optional, aber empfohlen: sie weist den Compiler an zu prüfen, ob die Methode wirklich eine geerbte Methode überschreibt. Tippfehler im Methodennamen werden so sofort als Compilerfehler erkannt.

Interfaces

Quelle: Skript Teil 1, Abschnitt „Interfaces”, S. 371–408

Grundlagen & Vererbung mit Interfaces

Ein Interface ist ein reiner Vertrag – eine Sammlung von Methodenköpfen ohne Implementierung. Klassen, die ein Interface implementieren, verpflichten sich, alle darin definierten Methoden zu implementieren. Der Vorteil: eine Klasse kann mehrere Interfaces implementieren (Java erlaubt nur Einfachvererbung bei Klassen).

// Interface-Definition
public interface Druckbar {
    void drucken();         // implizit public abstract
}

public interface Speicherbar {
    void speichern(String pfad);
}

// Klasse implementiert mehrere Interfaces
public class Bericht extends Dokument
        implements Druckbar, Speicherbar {

    @Override
    public void drucken() { System.out.println(”Drucke...”); }

    @Override
    public void speichern(String pfad) { /* ... */ }
}

Default-Methoden & mehrstufige Hierarchien

Seit Java 8 können Interfaces Default-Methoden enthalten – Methoden mit einem vorgegebenen Rumpf. Sie werden an implementierende Klassen vererbt und können optional überschrieben werden:

public interface Druckbar {
    void drucken(); // abstrakt – muss implementiert werden

    // Default-Methode – kann überschrieben werden, muss aber nicht
    default void druckenMitRahmen() {
        System.out.println(”--- Anfang ---”);
        drucken();
        System.out.println(”--- Ende ---”);
    }
}

Interfaces wirken sich auch auf Polymorphie aus: Eine Variable vom Interface-Typ kann jedes Objekt aufnehmen, dessen Klasse dieses Interface implementiert.

Prüfungshinweis Abstrakte Klasse vs. Interface: Abstrakte Klassen können Attribute und normale Methoden haben; eine Klasse kann nur von einer abstrakten Klasse erben. Interfaces dürfen keine Instanzvariablen haben (nur Konstanten); eine Klasse kann aber beliebig viele Interfaces implementieren. Wähle Interface, wenn eine reine Verhaltensvereinbarung ohne gemeinsame Attribute gefragt ist.

Exceptions

Zusammenfassung

Exceptions sind Objekte spezieller Klassen, die bei Fehler- oder Ausnahmesituationen erzeugt und „geworfen" werden. Sie ermöglichen eine saubere Trennung von normalem Standardcode (try-Block) und Fehlerbehandlungscode (catch-Block). Mit throw wirft eine Methode eine Exception; mit throws im Methodenkopf deklariert sie, dass sie eine Exception weiterreichen kann. Der finally-Block wird immer ausgeführt – egal ob eine Exception aufgetreten ist oder nicht.

Was sind Exceptions?

Quelle: Skript Teil 1, Abschnitt „Was sind Exceptions?", S. 411–413

Exceptions sind Objekte spezieller Klassen, die Informationen darüber enthalten, wo im Code ein Fehler aufgetreten ist. Sie können sehr schnell durch alle Aufrufebenen nach oben durchgereicht werden. Wichtige vorgefertigte Exception-Klassen:

  • NullPointerException – Zugriff auf eine null-Referenz
  • ArrayIndexOutOfBoundsException – Array-Index außerhalb des gültigen Bereichs
  • NumberFormatException – ungültiges Zahlenformat beim Parsen
  • FileNotFoundException – Datei nicht gefunden
  • IOException – allgemeine Ein-/Ausgabe-Fehler

Alle Exception-Klassen sind von der gemeinsamen Klasse Exception abgeleitet (und diese von Throwable).

Exceptions erzeugen und werfen: throw & throws

Quelle: Skript Teil 1, Abschnitt „Exceptions werfen", S. 414–424

throw und throws sind zwei verschiedene Schlüsselwörter mit unterschiedlicher Bedeutung:

  • throw steht im Methodenrumpf und wirft ein konkretes Exception-Objekt.
  • throws steht am Methodenkopf und deklariert, welche Exception-Typen die Methode weiterreichen kann.
public int returnZahlWert(char ziffer)
        throws NumberFormatException {   // Deklaration am Kopf

    if (ziffer >= '0' && ziffer <= '9') {
        return ziffer - '0';           // Normalfall
    } else {
        // Fehlfall: Exception-Objekt erzeugen und werfen
        NumberFormatException e =
            new NumberFormatException("Keine Ziffer!");
        throw e;
    }
}

Wenn eine aufrufende Methode eine throws-Methode aufruft, muss sie selbst entweder ebenfalls throws deklarieren oder die Exception mit try-catch fangen.

Exceptions fangen: try – catch – finally

Quelle: Skript Teil 1, Abschnitt „Exceptions fangen", S. 425–435

Mit try-catch wird eine Exception methodenintern gefangen – der Aufrufer merkt davon nichts und benötigt kein eigenes throws.

try {
    // "probiere" diesen Code – hier kann eine Exception entstehen
    String s = tastatur.next();
    char   c = s.charAt(0);
    int  wert = worker.returnZahlWert(c); // wirft ggf. Exception
}
catch (NumberFormatException ex) {
    // Single-Exception-Catch: gezielter Abfang eines Typs
    ex.printStackTrace();
}
catch (StringIndexOutOfBoundsException ex) {
    System.out.println("Leere Eingabe!");
}
catch (Exception ex) {
    // Allgemeiner Catch – IMMER als letzter Block!
    System.out.println("Unbekannter Fehler: " + ex.getMessage());
}
finally {
    // Wird IMMER ausgeführt – ideal zum Schließen von Ressourcen
    System.out.println("Fertig.");
}

Alternativ: Multiple-Exception-Catch – mehrere Typen in einem Block mit | verbunden:

catch (NumberFormatException | StringIndexOutOfBoundsException ex) {
    ex.printStackTrace(); // gleiche Behandlung für beide Typen
}
Prüfungshinweis Der finally-Block ist optional, wird aber – wenn vorhanden – immer ausgeführt (auch wenn kein Fehler aufgetreten ist). Er eignet sich zum Schließen von Streams oder Datenbankverbindungen. Der catch(Exception e)-Block muss immer als letzter stehen, weil er alle Ausnahmen abfängt – andernfalls entsteht ein Compilerfehler.

Einstieg Streams

Zusammenfassung

Streams sind Datenflüsse zwischen einer Quelle (z. B. Tastatur, Datei, Netzwerk) und einem Ziel (z. B. Bildschirm, Datei). Java unterscheidet zeichenorientierte Streams (für Textdaten) und byteorientierte Streams (für Binärdaten). Streams werden durch Verschachtelung der Konstruktoren kombiniert. Da Stream-Methoden Exceptions werfen, muss immer IOException behandelt werden. Geöffnete Streams müssen mit close() wieder geschlossen werden.

Das Stream-Konzept in Java

Quelle: Skript Teil 1, Abschnitt „Stream-Konzept", S. 437–447

Bisher wurde der Scanner für Tastatureingaben genutzt. Streams sind leistungsfähiger und flexibler: Sie können dieselben Daten aus beliebigen Quellen lesen – Tastatur, Datei, Netzwerkverbindung. Das Prinzip ist immer gleich: Streams werden so verschachtelt, dass die gewünschte Datenquelle mit der gewünschten Konvertierung kombiniert wird:

import java.io.*; // alle Stream-Klassen aus java.io

// Tastatureingabe zeilenweise lesen – Verschachtelung der Konstruktoren:
BufferedReader konsole = new BufferedReader(
    new InputStreamReader(System.in));

String eingabe = konsole.readLine(); // liest eine Zeile

Der InputStreamReader konvertiert die Rohbytes des Tastaturpuffers in Zeichen; der BufferedReader puffert und ermöglicht die zeilenweise Verarbeitung mit readLine().

Vorgehen bei der Nutzung von Streams:

  1. Klassen mit import java.io.*; einbinden.
  2. Exceptions behandeln: entweder throws IOException am Methodenkopf oder intern mit try-catch.
  3. Streams kombinieren: Datenquelle + Konvertierung verschachteln.
  4. Stream nach Gebrauch mit close() schließen.

Dateizugriff mit zeichenorientierten Streams

Quelle: Skript Teil 1, Abschnitt „Dateizugriff mit zeichenorientierten Streams", S. 448–452

Für textbasierte Dateizugriffe werden FileReader/BufferedReader (Lesen) und FileWriter/BufferedWriter (Schreiben) verwendet:

// ---- Datei lesen ----
public String dateiEinlesen(String dateiname)
        throws IOException, FileNotFoundException {

    String inhalt = "", zeile;
    BufferedReader leser = new BufferedReader(
        new FileReader(dateiname));

    // readLine() gibt null zurück, wenn das Dateiende erreicht ist
    while ((zeile = leser.readLine()) != null) {
        inhalt += zeile + "\n"; // readLine() schneidet \n ab → wieder anfügen
    }
    leser.close(); // Stream MUSS geschlossen werden!
    return inhalt;
}

// ---- Datei schreiben ----
public void dateiSchreiben(String dateiname, String inhalt)
        throws IOException {

    BufferedWriter schreiber = new BufferedWriter(
        new FileWriter(dateiname));

    schreiber.write(inhalt); // gesamten String schreiben
    schreiber.close();       // Stream schließen – schreibt gepufferte Daten heraus
}
Prüfungshinweis Wenn ein Stream nicht mit close() geschlossen wird, kann die Zieldatei leer oder unvollständig sein – gepufferte Daten wurden noch nicht herausgeschrieben. readLine() schneidet automatisch den Zeilenumbruch ab; dieser muss beim Aufbau des Rückgabe-Strings explizit mit + "\n" wieder hinzugefügt werden.

Der Klassenzoo der Streams

Quelle: Skript Teil 1, Abschnitt „Klassenzoo der Streams", S. 453–454

Java teilt Streams in zwei große Familien:

Familie Basisklassen Zweck Wichtige Klassen
Zeichenorientiert Reader / Writer Textdateien, Tastatur, Zeichenketten FileReader, BufferedReader, InputStreamReader, FileWriter, BufferedWriter, PrintWriter
Byteorientiert InputStream / OutputStream Binärdaten: Bilder, Audio, Objekte FileInputStream, DataInputStream, ObjectInputStream, FileOutputStream, DataOutputStream, ObjectOutputStream

Die Verschachtelung der Konstruktoren folgt immer dem gleichen Muster: Innerster Konstruktor = Datenquelle/Datensenke; äußere Konstruktoren = Konvertierung und Pufferung.

Programmieren im Großen – Packages

Zusammenfassung

Packages strukturieren Programmsysteme oberhalb der Klassenebene. Zusammengehörende Klassen werden in gemeinsamen Unterverzeichnissen zusammengefasst. Jede Member-Klasse deklariert ihre Zugehörigkeit in der ersten Zeile mit der package-Anweisung; der Java-Compiler legt Compilate (.class-Dateien) automatisch im passenden Unterverzeichnis ab.

Wozu Packages?

Quelle: Skript Teil 2, Abschnitt „Wozu Packages?”, S. 466–467

Große, reale Programmsysteme enthalten oft mehrere hundert Klassen. Ohne übergeordnete Strukturierung wäre ein solches System kaum wartbar. Packages lösen dieses Problem:

  • Packages ordnen Gruppen fachlich zusammengehörender Klassen in gemeinsame Unterverzeichnisse ein.
  • Eine Klasse, die zu einem Package gehört, heißt Member dieses Packages.
  • Die Zugehörigkeit ist direkt im Quelltext jeder Member-Klasse vermerkt.

Zuordnung von Klassen zu Packages

Quelle: Skript Teil 2, Abschnitt „Zuordnung von Klassen zu Packages”, S. 468–471

Einfache Packages

Die package-Anweisung steht als erste Zeile vor dem Klassenkopf. Java legt automatisch einen Unterordner mit dem Package-Namen an:

package mypackage;
public class MeineKlasse {
    // Compilat landet in ./mypackage/MeineKlasse.class
}

Verschachtelte (Sub-)Packages

Hierarchiestufen werden durch . getrennt – dieser wird plattformabhängig zum Verzeichnistrenner aufgelöst (Linux: /, Windows: \):

package mypackage.sub1.sub2;
public class MeineKlasse {
    // Compilat: ./mypackage/sub1/sub2/MeineKlasse.class
}

Import von Klassen aus fremden Packages

Quelle: Skript Teil 2, Abschnitt „Import von Klassen aus fremden Packages”, S. 472–473

import-Anweisungen stehen nach der package-Zeile, vor dem Klassenkopf. Es gibt zwei Varianten:

// Einzelne Klasse importieren:
import java.io.BufferedReader;

// Alle Klassen eines Packages importieren (Wildcard):
import java.io.*;
Prüfungshinweis Der Wildcard-Import import java.io.* bindet nur die direkten Klassen des genannten Packages ein – nicht die Klassen eventueller Sub-Packages! Eingebunden werden immer Compilate (.class-Dateien), keine Quelltexte.

Compilierung und Klassenpfad

Quelle: Skript Teil 2, Abschnitt „Compilierung und Klassenpfad”, S. 474–479

Empfohlene Verzeichnisstruktur: Quelldateien in src/, Compilate in bin/. Wichtige Compiler-Optionen:

Option Bedeutung
-cp <pfad> Klassenpfad: Wo liegen importierte Klassen? Mehrere Pfade mit ; (Win) oder : (Linux) trennen.
-d <pfad> Destination: Zielverzeichnis für Compilate. Package-Unterordner werden automatisch angelegt. 
// Aufruf aus dem Elternverzeichnis von src und bin:
javac -cp ”.\bin” -d ”.\bin” .\src\mypackage\Klasse1.java

In Eclipse werden Packages über File → New → Package angelegt. Eclipse setzt die package-Anweisung und verwaltet den Klassenpfad automatisch.

Packages in der UML

Quelle: Skript Teil 2, Abschnitt „Darstellung von Packages in der UML”, S. 480–481

Packages können in zwei UML-Diagrammarten dargestellt werden:

  • Klassendiagramm mit Packages: Klassen werden innerhalb eines Package-Rahmens (mit Namens-Tab oben links) gezeigt. <<use>>-Abhängigkeiten zwischen Packages kennzeichnen Nutzungsbeziehungen.
  • Paketdiagramm: Zeigt nur die Package-Struktur und Enthält-Beziehungen. Innere Packages werden als ineinandergeschachtelte Rechtecke dargestellt.
Prüfungshinweis Enthält-Beziehungen im Paketdiagramm sehen ähnlich aus wie Aggregation/Komposition im Klassendiagramm – aber Kontext ist entscheidend: Im Paketdiagramm steht ein Package im anderen, nicht ein Objekt in einem anderen Objekt.

Streams – Binär & Objektserialisierung

Zusammenfassung

Aufbauend auf den zeichenorientierten Streams (Teil 1) kommen zwei neue Stream-Typen hinzu: Binärstreams für nicht-textuelle Daten (Bilder, Videos) und Objektstreams für die Serialisierung ganzer Java-Objekte als Byte-Sequenz. Klassen, deren Objekte serialisiert werden sollen, müssen das Marker-Interface Serializable implementieren.

Binärorientierte Streams

Quelle: Skript Teil 2, Abschnitt „Dateizugriff mit einfachen, byteorientierten Streams”, S. 515–523

Binärstreams verarbeiten rohe Byte-Daten und sind von InputStream (Lesen) bzw. OutputStream (Schreiben) abgeleitet. Wie bei zeichenorientierten Streams werden sie bei der Erzeugung nach Bedarf verschachtelt:

// Binäres Kopierprogramm – verarbeitet 4 Bytes pro Schritt
byte[] dateibytes = new byte[4];

DataInputStream quelle = new DataInputStream(
    new FileInputStream(args[0]));
DataOutputStream senke = new DataOutputStream(
    new FileOutputStream(args[1]));

int anzBytes;
while ((anzBytes = quelle.read(dateibytes)) != -1) {
    senke.write(dateibytes, 0, anzBytes);
}
quelle.close(); // Streams MÜSSEN nach Gebrauch geschlossen werden!
senke.close();
Prüfungshinweis read() gibt −1 zurück, wenn das Ende des Streams erreicht ist. Streams müssen mit close() geschlossen werden – sonst kann die Zieldatei leer oder unvollständig sein, da gepufferte Daten nicht herausgeschrieben werden.

Objektserialisierung: ObjectOutputStream & ObjectInputStream

Quelle: Skript Teil 2, Abschnitt „Verarbeitung von Java-Objekten mit Streams”, S. 524–543

ObjectOutputStream und ObjectInputStream können ganze Java-Objekte als Einheit schreiben bzw. lesen. Das Schreiben heißt serialisieren, das Lesen deserialisieren.

Voraussetzung: Die betroffene Klasse muss implements Serializable deklarieren. Serializable ist ein leeres Marker-Interface – es enthält keine Methoden, markiert aber die Klasse als serialisierbar.

import java.io.Serializable;

public class BenzinVerbrauch implements Serializable {
    private double literGetankt;
    private String  datum;
    private int    kmStand;
    // Konstruktor, getter/setter ...
}

// ---- Serialisieren (Schreiben) ----
ObjectOutputStream out = new ObjectOutputStream(
    new FileOutputStream(”Fahrtenbuch.tmp”));
out.writeInt(2);             // Anzahl voranstellen
out.writeObject(verbrauch1); // Objekte schreiben
out.writeObject(verbrauch2);
out.close();

// ---- Deserialisieren (Lesen) ----
ObjectInputStream in = new ObjectInputStream(
    new FileInputStream(”Fahrtenbuch.tmp”));
int anzahl = in.readInt();
for (int i = 0; i < anzahl; i++) {
    // readObject() liefert Object → TypeCast nötig
    BenzinVerbrauch b = (BenzinVerbrauch) in.readObject();
}
in.close();
Prüfungshinweis ClassNotFoundException wird beim Deserialisieren geworfen, wenn die Ziel-Klasse im Programm nicht existiert oder beim Schreiben und Lesen unterschiedliche Package-Pfade verwendet wurden. Immer im catch abfangen!

Generische Klassen (Generics)

Zusammenfassung

Generics sind typsichere Behälterklassen für beliebige Referenztypen. Der Typ-Parameter <T> wird bei der Instanziierung mit einem konkreten Datentyp belegt. So entfallen unsichere Casts auf Object. Das gesamte Java-Collection-Framework (ArrayList, HashMap, HashSet …) baut auf Generics auf.

Wozu Generics?

Quelle: Skript Teil 2, Abschnitt „Wozu Generics?”, S. 546–549

Generics sind Behälter für Referenzvariablen beliebigen Typs – z. B. ArrayList, Vektoren, Sets und Maps. Der Typ wird über spitze Klammern übergeben:

// Bekannt: ArrayList mit Typ-Parameter
ArrayList<String> liste = new ArrayList<String>();

// Verallgemeinerte Form jeder Generic:
MeinGeneric<Typ> behaelter = new MeinGeneric<Typ>();

Bisher wurden Daten-Parameter über runde Klammern () durchgereicht. Bei Generics kommt ein Typ-Parameter in spitzen Klammern <> hinzu, der bei Deklaration und Konstruktoraufruf angegeben wird.

Einfache Generics – Beispiel Couple<T>

Quelle: Skript Teil 2, Abschnitte „Einfache Generics” & „Standard-Interfaces”, S. 550–559

Eine Generic-Klasse deklariert ihren Typ-Parameter im Klassenkopf. Alle Vorkommen von T innerhalb der Klasse werden bei der Instanziierung durch den konkreten Typ ersetzt:

public class Couple<T> {

    private T member1;
    private T member2;

    public Couple(T member1, T member2) {
        this.member1 = member1;
        this.member2 = member2;
    }

    public void print() {
        System.out.println(”Member 1: “ + member1.toString());
        System.out.println(”Member 2: “ + member2.toString());
    }
}

// Verwendung mit verschiedenen Typen:
Couple<Studierende> team = new Couple<Studierende>(s1, s2);
Couple<String>      duo  = new Couple<String>(”Asterix”, ”Obelix”);

toString() ist immer aufrufbar, da jede Klasse von Object erbt und damit mindestens die rudimentäre Object.toString() verfügbar ist. Eigene Klassen wie Person können sie überschreiben.

Standard-Interfaces für Generics

  • Iterable<T>: Methode iterator() – ermöglicht die Iterator-Schleife for(T x : generic).
  • Comparable: Methode compareTo() – ermöglicht Vergleiche für Sortierungen. Comparable ist selbst eine Generic.

Iterierbare Generics

Quelle: Skript Teil 2, Abschnitt „Iterierbare Generics”, S. 560–599

Eine iterierbare Generic implementiert Iterable<T> und stellt über deren Methode iterator() ein Iterator-Objekt bereit. Dieser Iterator implementiert Iterator<T> mit den Methoden hasNext() und next(). Hinter jedem for(T x : behälter)-Loop verbirgt sich genau dieser Mechanismus.

import java.util.Iterator;

public class SmallList<T> implements Iterable<T> {

    private Object[] values;   // T[] ist vom Compiler nicht erlaubt!
    private int noOfElements;

    public SmallList(int maxSize) {
        values = new Object[maxSize];
        noOfElements = 0;
    }

    public void add(T element) {
        // bei Bedarf: Arrays.copyOf(values, values.length + 1)
        values[noOfElements++] = element;
    }

    public T get(int index) {
        return (T) values[index]; // TypeCast auf T nötig
    }

    @Override
    public Iterator<T> iterator() {
        return new SmallListIterator<T>(this);
    }
}
Prüfungshinweis T[] values = new T[size] wird vom Compiler nicht akzeptiert – stattdessen Object[] verwenden, beim Zugriff TypeCast auf (T). Verlängerung: Arrays.copyOf(values, values.length + 1). Teilkopie: Arrays.copyOfRange(values, von, bis).

Wildcards bei der Typisierung

Quelle: Skript Teil 2, Abschnitt „Wildcards bei der Typisierung von Generics”, S. 600–626

Wildcards ermöglichen Methoden, eine Generic mit zur Programmierzeit unbekanntem Inhaltstyp entgegenzunehmen:

Syntax Beschreibung Typischer Zweck
Generic<?> Beliebiger Typ Nur lesen; kein setMember() möglich
Generic<? extends Clazz> Clazz oder Kindklasse (Upper-Bound) Lesen – Typ ist garantiert mindestens Clazz
Generic<? super Clazz> Clazz oder Superklasse (Lower-Bound) Schreiben – Objekte vom Typ Clazz sicher einfügen 
// Upper-bound: Liste lesen – enthält Person oder Kindklasse
public void printPersons(ArrayList<? extends Person> liste) {
    for (Person p : liste) {
        System.out.println(p.getVorname() + ” “ + p.getNachname());
    }
}

// Lower-bound: in Liste schreiben – nimmt RennTeilnehmer auf
public void addCandidates(
        ArrayList<? super RennTeilnehmer> ziel,
        ArrayList<? extends RennTeilnehmer> quelle) {
    for (RennTeilnehmer r : quelle) ziel.add(r);
}

Das Java-Collection-Framework

Quelle: Skript Teil 2, Abschnitt „Das Java-Collection-Framework”, S. 627–685

Java-Collections sind vorgefertigte generische Datenstrukturen. Alle implementieren eine Teilmenge der Interfaces Iterable, Collection, List, Set, Queue oder Map.

ArrayList – geordnete, wahlfreie Liste

  • Geordnet (Reihenfolge des Einfügens), wahlfrei via get(int index).
  • Nicht thread-sicher, Duplikate erlaubt.
  • Wichtigste Methoden: add(), remove(Object), remove(int), get(), size(), clear(), isEmpty(), sort(Comparator).
// Comparable in der Datenklasse implementieren:
public int compareTo(Object o) {
    Person p = (Person) o;
    return this.persnr - p.getPersnr(); // <0 | 0 | >0
}

// Comparator-Klasse für sort():
public class ComparePerson implements Comparator<Person> {
    @Override
    public int compare(Person p1, Person p2) {
        return p1.compareTo(p2);
    }
}
liste.sort(new ComparePerson()); // nach Persnr. sortieren

HashMap<K, V> – Schlüssel-Wert-Zuordnung

  • Objekte (Values) werden nach einem eindeutigen Schlüssel (Key) abgelegt.
  • Keys sind nicht duplizierbar, Values sind duplizierbar.
  • Nicht direkt iterierbar; Zugriff über Key: get(key).
  • Methoden: put(k,v), get(k), remove(k), replace(k, vAlt, vNeu), containsKey(k), containsValue(v).
HashMap<Integer, Person> map = new HashMap<>();
map.put(p1.getPersnr(), p1);   // Key = Personalnummer (Integer)
Person p = map.get(5);         // Direktzugriff per Key
map.remove(3);                  // Eintrag mit Key 3 löschen
map.replace(3, pAlt, pNeu);    // Value tauschen

HashSet<T> – Menge ohne Duplikate

  • Keine Duplikate – standardmäßig werden Objektreferenzen verglichen.
  • Ist iterierbar; Methoden: add(), remove(), clear(), size().
Prüfungshinweis Zwei verschiedene Referenzen mit identischem Inhalt gelten im HashSet als unterschiedliche Elemente, solange keine angepasste equals()-Methode implementiert ist. Nur das nochmalige Einfügen der gleichen Referenz wird als Duplikat erkannt und abgewiesen.

Annotationen

Zusammenfassung

Annotationen hängen Metainformation an Quellcode-Elemente (Klassen, Methoden, Attribute). Sie beginnen mit @, werden vom Compiler ausgewertet und können selbst definiert werden – über @interface. Java liefert vorgefertigte Annotationen wie @Override, @Target oder @Retention. Selbstdefinierte Annotationen können über die Java-Reflections-API zur Laufzeit ausgelesen werden.

Was sind Annotationen?

Quelle: Skript Teil 2, Abschnitt „Annotationen”, S. 687–692

Eine Annotation ist Metainformation, die über die Java-Modifizierer hinausgeht. Sie wird als Interface mit dem Schlüsselwort @interface definiert und kann attributähnliche Methoden enthalten:

// Definition einer Marker-Annotation (ohne Attribute):
public @interface SoDa { }

// Verwendung – ohne Einschränkung an Klasse und Methode:
@SoDa
public class MeineKlasse {
    @SoDa
    public void hallo() {
        System.out.println(”Ich bin einfach so da!”);
    }
}

Erlaubte Attributtypen in Annotationen: String, elementare Datentypen (int, nicht Integer), Class, andere Annotationen, Enum-Typen sowie eindimensionale Arrays der genannten Typen.

Eigene Annotationen mit API-Annotationen konfigurieren

Quelle: Skript Teil 2, Abschnitt „Weitere Eigenschaften von eigenen Annotationen”, S. 693–701

Eigene Annotationen werden selbst mit API-Annotationen präzisiert:

API-Annotation Bedeutung
@Target(TYPE) Darf nur an Klassen-/Interface-Deklarationen gehängt werden.
@Target(METHOD) Darf nur an Methoden-Köpfe gehängt werden.
@Retention(RUNTIME) Annotation ist auch zur Laufzeit über Reflections abfragbar.
@Documented Erscheint in der Javadoc-Dokumentation.
@Inherited Wirkt automatisch auch auf abgeleitete Klassen. 
import static java.lang.annotation.ElementType.TYPE;
import static java.lang.annotation.RetentionPolicy.RUNTIME;
import java.lang.annotation.Retention;
import java.lang.annotation.Target;

@Retention(RUNTIME)
@Target(TYPE)
public @interface ClassHeader {
    int    version();   // Attributmethoden
    String faculty();
    String university();
}

// Verwendung (Attributwerte als benannte Parameter):
@ClassHeader(version=1, faculty=”Informatik”, university=”HSAS”)
public class MeineKlasse { ... }

Laufzeitabfrage via Java-Reflections

Quelle: Skript Teil 2, Abschnitt „Laufzeit-Abfrage der angehefteten Annotationen”, S. 702–713

Java-Reflections ermöglichen zur Laufzeit Abfragen über Klassen, Methoden und deren Annotationen. Zentral ist die statische Variable .class jeder Java-Klasse vom Typ java.lang.Class:

import java.lang.reflect.Method;

// Annotation an Klasse prüfen und auslesen:
if (MeineKlasse.class.isAnnotationPresent(ClassHeader.class)) {
    ClassHeader ch =
        MeineKlasse.class.getAnnotation(ClassHeader.class);
    System.out.println(”Faculty: “ + ch.faculty());
}

// Alle Methoden einer Klasse nach Annotation durchsuchen:
for (Method m : MeineKlasse.class.getMethods()) {
    if (m.isAnnotationPresent(MethodHeader.class)) {
        MethodHeader mh = m.getAnnotation(MethodHeader.class);
        System.out.println(m.getName() + ” – Autor: “ + mh.author());
    }
}
Prüfungshinweis Annotationen sind zur Laufzeit nur abfragbar, wenn sie mit @Retention(RUNTIME) gekennzeichnet sind. Ohne diese Angabe sind sie nur zur Compile-Zeit verfügbar und für Reflections unsichtbar.

Threads

Zusammenfassung

Threads sind parallel bzw. quasi-parallel ausführbare Bestandteile eines Prozesses. In Java werden sie durch Objekte der Klasse Thread oder durch Klassen, die Runnable implementieren, realisiert. Der gleichzeitige Zugriff mehrerer Threads auf gemeinsame Daten muss durch synchronized-Methoden oder -Blöcke geschützt werden (Monitor-Konzept).

Threads vs. Prozesse

Quelle: Skript Teil 2, Abschnitt „Threads vs. Prozesse”, S. 716–717

Ein Prozess ist ein Programm in Ausführung (mit eigener PID). Ein Thread ist ein parallel ausführbarer Bestandteil innerhalb eines Prozesses – mehrere Threads teilen sich den Adressraum des Prozesses.

Typische Anwendungsfälle:

  • Server bedient mehrere Clients gleichzeitig – für jede Client-Anfrage ein eigener Thread.
  • Große Bildmatrix wird in Kacheln zerlegt und jede Kachel von einem separaten Thread verarbeitet.
  • Producer-Consumer: ein Thread erzeugt Datensätze, ein anderer verarbeitet sie sobald sie verfügbar sind.

Threads in Java – zwei Architekturen

Quelle: Skript Teil 2, Abschnitt „Threads in Java”, S. 718–745

Weg 1: Klasse extends Thread

public class MeinThread extends Thread {
    @Override
    public void run() {
        // nebenlaeufiger Code
    }
}
MeinThread t = new MeinThread();
t.start(); // startet run() als Nebenläufigkeit in der VM

Nachteil: Die Klasse kann keine eigene weitere Superklasse haben, da die Superklassenbeziehung für Thread benötigt wird.

Weg 2: Klasse implements Runnable (empfohlen)

// Eigene Vererbungshierarchie bleibt frei:
public class Gast extends Person implements Runnable {
    @Override
    public void run() {
        // nebenlaeufiger Code des Gastes
    }
}

Gast   g = new Gast(”Emil”, huette);
Thread t = new Thread(g); // Runnable an Thread übergeben
t.start();                  // startet g.run() als Nebenläufigkeit
Prüfungshinweis Das Erzeugen eines Thread-Objektes startet es noch nicht – erst t.start() aktiviert die Nebenläufigkeit. Ein direkter Aufruf von t.run() wäre nur ein normaler sequentieller Methodenaufruf, keine Parallelisierung!

Thread-Synchronisation – das Monitor-Konzept

Quelle: Skript Teil 2, Abschnitt „Thread-Synchronisation”, S. 746–774

Wenn mehrere Threads gleichzeitig schreibend auf dasselbe Objekt zugreifen, entstehen kritische Abschnitte, die zu inkonsistenten Zuständen oder Exceptions führen können. Java löst dies mit dem Schlüsselwort synchronized: Nur ein Thread darf gleichzeitig im synchronisierten Bereich sein – das Objekt wird zum Monitor, wartende Threads werden in eine Warteschlange eingestellt.

Drei Arten der Synchronisation

// 1. Synchronisierte Instanz-Methode → Objekt ist Monitor
public synchronized int kochen(String name, int gramm) {
    // geschützter Code – nur ein Thread gleichzeitig
}

// 2. Synchronisierte statische Methode → Klasse ist Monitor
public static synchronized int kochen(...) { ... }

// 3. Synchronisierter Block auf Referenz → Referenz ist Monitor
synchronized (herd) {
    // geschützter Code bezogen auf das herd-Objekt
}

Thread-sichere Collections

Die meisten Java-Collections sind nicht thread-sicher. Sichere Alternativen:

  • Vector statt ArrayList
  • ConcurrentHashMap statt HashMap
Prüfungshinweis ArrayList und HashMap sind nicht thread-sicher! Parallele schreibende Zugriffe können zu NullPointerException oder korrumpierten Daten führen – wie im Spaghetti-Hütten-Beispiel gezeigt. Kritische Abschnitte mit synchronized schützen oder auf Vector/ConcurrentHashMap ausweichen.
